Enterprise Treasury Intelligence Platform

LLD v2.0  June 2026 — Internal Engineering

This document specifies the complete low-level technical design for a dual-lane treasury intelligence platform. It ingests BAI2/EDI files, PDFs, CSVs, and intraday Kafka feeds; extracts structured financial data; computes bond and treasury analytics; persists across a hybrid storage tier; and serves a commercial AI chatbot answering 60+ query types with a 15-second SLA and 100% numerical accuracy guarantee.

1. System Overview

The platform processes enterprise treasury documents and serves a conversational AI interface over the resulting structured data. 70% of queries are answered via deterministic NL2SQL + indexed PostgreSQL, 20% via hybrid RAG, and 10% via the bond calculation engine. The LLM is used only for language generation, never for arithmetic.

3. Ingestion & Extraction Pipeline

3.1 Document Classifier

Every uploaded file is routed before any parsing begins using MIME detection and content-level header matching (BAI2 record type 01, EDI ISA* header).

3.2 BAI2 Structured Parser

Parses all BAI2 record codes (01/02/03/16/49/98/99). Normalises amounts from cents to dollars. Maps 100+ type codes to semantic categories: wire_in, wire_out, ach_receive, ach_origination, lockbox, check, fee, return, credit, debit. Determines settlement status from the funds_type field.

3.3 PDF/Image Lane — OCR + NER

Uses Unstructured.io with YOLOX layout detection for high-fidelity table extraction from bond statements and bank reports. LayoutLMv3 fine-tuned model extracts bond entities (ISIN, coupon rate, maturity, settlement price, accrued interest).

3.4 Vendor Entity Resolver

Resolves noisy BAI2 description fields (ACH credit strings, SWIFT BNF/ORG fields, lockbox IDs) against the ERP vendor master using rapidfuzz token_set_ratio with an 82-point threshold and four regex patterns.

3.5 Celery Async Pipeline

Structured lane: parse → vendor resolve → validate → cache invalidate. Unstructured lane: OCR → NER → bond calc → validate → embed + index. Both lanes use Celery task chains with max_retries=3 and 300-second time limits.

4. Storage Layer

4.1 PostgreSQL Schema

Core tables: bank_accounts, account_balances, transactions, bonds, bond_positions. Two materialized views refresh on ingestion completion: daily_account_summary (credits, debits, wire counts, pending counts) and portfolio_summary (DV01, duration, market value). Key composite indexes on (account_id, as_of_date), (as_of_date, type_category), amount DESC, vendor_customer, and settlement_type.

4.2 Qdrant Vector Store

Collection treasury_docs with dual vectors: dense (BGE-M3, 1024-dim, cosine) and sparse (BM25 via fastembed). HNSW m=32, ef_construct=200. Payload indexes on portfolio_id, isin, account_id, doc_type, as_of_date for pre-filtering before vector search.

4.3 Redis Layered Cache

L1 in-process LRU (1024 entries, microseconds) for hot embeddings. L2 Redis Sentinel with per-prefix TTLs: SQL results 300 s, balance 60 s, embeddings 3600 s. Cache keys are SHA-256 hashes of the normalised query payload. On ingestion completion, account-scoped cache keys are invalidated.

5. Query Routing & NL2SQL Engine

5.1 Intent Router

First attempts regex-based template matching against 30 known patterns. Template-matched queries bypass the NL2SQL LLM entirely and execute pre-compiled parameterised SQL in under 50 ms. Unknown queries are classified by the LLM into 13 intents and routed to the appropriate path.

5.2 NL2SQL Engine

Novel queries go to Qwen2.5-Coder-7B-Instruct via vLLM with a 3-second hard timeout. The system prompt constrains the model to the exact schema and forbids mutating statements. A safety regex blocks DROP, DELETE, UPDATE, INSERT, TRUNCATE, ALTER, CREATE, GRANT, EXEC. sqlparse validates output before execution.

6. RAG Retrieval Layer

Hybrid retrieval fuses dense (BGE-M3 cosine) and sparse (BM25) results via Reciprocal Rank Fusion in Qdrant. The top-20 candidates are re-ranked by a CrossEncoder (ms-marco-MiniLM-L-6-v2) to return the best 6 chunks. Payload filters restrict retrieval to the relevant portfolio, account, or document type before vector search executes, keeping latency under 500 ms.

7. Bond Calculation Engine

All treasury math is deterministic Python (scipy + numpy). The LLM never computes these values — it only formats and narrates them.

8. Numerical Accuracy Guarantee — 4-Layer Defence

  1. SQL-first: All figures are pre-computed by PostgreSQL or BondCalculator and injected as literals into the prompt.
  2. Structured output: The LLM returns Pydantic InsightResponse with each claim tagged by source (database / calc_engine / llm_inferred).
  3. Numerical Verifier: Any llm_inferred figure is auto-corrected from the ground-truth dict. Relative error >0.01% triggers a correction and increments a Prometheus counter.
  4. temperature=0.0: All financial generation calls use deterministic sampling.

12. Observability

OpenTelemetry traces every ingestion and query path end-to-end. Prometheus counters and histograms track: documents ingested by route and status, OCR latency, NL2SQL latency by path (template vs LLM), SQL execution latency by query type, LLM token counts and generation latency, SLA breach count, cache hit rate by prefix, verifier corrections by field, and active chatbot sessions. Grafana dashboards surface system health and data accuracy in a single view.
